## Challenges

### Starter Template Surprises

The `with-supabase` starter had several undocumented behaviors that consumed time across multiple stories:

- pnpm not pre-installed (Story 1.1)
- `create-next-app` rejected capital letters in project name — required rsync workaround (Story 1.1)
- Starter uses flat directory structure, not `src/` (Story 1.1) — all architecture references needed adapting
- `geist` package not pre-installed despite being listed as a starter dependency (Story 1.1)
- shadcn already initialized — required checking before re-running init (Story 1.1)
- `proxy.ts` naming — Next.js middleware must be `middleware.ts` at root; proxy.ts is a utility, not the entry point (Story 1.1 review)
- `.env.example` had wrong Supabase key name (`ANON_KEY` vs `PUBLISHABLE_KEY`) (Story 1.1 review)

### Recurring Code Review Patterns

These classes of bugs appeared across multiple stories and should be proactively avoided going forward:

1. **Whitespace validation** — `password.trim()` used for length check but raw (untrimmed) value sent to API (Stories 1.3, 1.5)
2. **Double-submit guard placement** — `if (isLoading) return` missing or placed after state mutation (Stories 1.3, 1.4)
3. **Silent error handling** — Supabase results destructured but error branch not handled, or error discarded entirely (Stories 1.3, 1.4, 1.5)
4. **Router navigation** — `router.push()` without `router.refresh()` after auth state changes; `router.push()` instead of `router.replace()` after one-time operations (Story 1.4, 1.5)
5. **Server Action form pattern** — fire-and-forget call from event handler vs. proper `<form action={serverAction}>` pattern (Story 1.4)

### Technical Debt Accumulated

See action items below for disposition of each deferred item.

## Key Insights

1. **The code review step is essential** — it caught issues the implementation pass missed on every story. The review-then-patch cycle works; keep it.
2. **Spec ≠ corrected behavior** — story dev notes preserve the original spec, not the patched outcome. Future agents reading dev notes as canonical patterns will re-introduce fixed bugs. `project-context.md` is the right place to document learned corrections.
3. **Starter template assumptions should be validated at story 1.1** — future projects should do a starter audit as the first task, not discover surprises per story.
4. **Deferred items compound** — the `updated_at` trigger gap, open redirect, and dark mode issue each touch multiple files and each future story that touches those files risks inheriting the gap.

## project-context.md Updates

The following corrected patterns from Epic 1 code reviews were added to `project-context.md` to prevent regression in future stories:

- Canonical browser Supabase client is `createBrowserClient()` from `@/lib/supabase/browser` (not `@/lib/supabase/client`)
- Server Actions that trigger navigation use `<form action={serverAction}>` pattern
- `router.refresh()` required before `router.push()` when destination reads session state
- `router.replace()` (not `push()`) after destructive/one-time auth operations
- Password/text input: trim before validate AND send trimmed value to API
- `updated_at` must be set explicitly in all UPDATE statements (no auto-trigger exists)
- Midgard is dark-only — ThemeProvider must use `defaultTheme="dark"` + `forcedTheme="dark"`
- Middleware public path allowlist must be updated in BOTH the redirect block AND the `isPublicPath` check in `proxy.ts` when adding new public routes
